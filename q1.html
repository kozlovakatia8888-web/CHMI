🌐 Архитектурные парадигмы веб-фронтенда: MPA, SPA, SSR, SSG

Полный разбор с примерами, плюсами/минусами, таблицей сравнения и тестом для самопроверки.

📌 Зачем разбираться в парадигмах?

Выбор архитектуры определяет, как быстро загрузится сайт, насколько он будет дружелюбен к поисковым роботам и насколько удобен для пользователя. Каждая парадигма — это компромисс между скоростью первого рендера, интерактивностью и сложностью разработки.

1. MPA (Multi‑Page Application)

Принцип: при каждом клике по ссылке сервер генерирует и отправляет новый HTML-документ. Страница перезагружается полностью.

// Пример на Express.js (сервер рендерит HTML) app.get('/about', (req, res) => { res.send('<html><body><h1>О нас</h1></body></html>'); });
  • Рендеринг: полностью на сервере.
  • SEO: идеальное — поисковый робот видит готовый HTML.
  • TTI (Time to Interactive): быстрый первый показ, но навигация требует перезагрузки.
  • Плюсы: простота, надёжность, кеширование на уровне CDN.
  • Минусы: перезагрузка страницы при любом действии, хуже пользовательский опыт при сложных интерфейсах.
📌 Примеры MPA: корпоративные сайты, новостные порталы, WordPress без плагинов-одностраничек.

2. SPA (Single‑Page Application)

Принцип: при первом визите загружается "скелет" HTML и большой JS-бандл. Далее роутинг на клиенте: подгружаются только данные через API, страница не перезагружается.

// Пример React с React Router function App() { return ( <Router> <Link to="/about">О нас</Link> <Route path="/about" component={About} /> </Router> ); }
  • Рендеринг: инициализация на сервере, отрисовка на клиенте (CSR).
  • SEO: плохое без дополнительных ухищрений (из-за пустого <div id="root"></div>).
  • TTI: медленный первый запуск (загрузка и выполнение JS), затем мгновенная навигация.
  • Плюсы: плавность, богатые интерфейсы, меньше трафика при переходах.
  • Минусы: сложность, SEO требует серверного рендеринга или prerendering.
📌 Примеры SPA: Gmail, Figma, Trello, внутренние админки.

3. SSR (Server‑Side Rendering) с гидратацией

Принцип: первый запрос — сервер генерирует полный HTML (как MPA), параллельно загружается JS-бандл. После загрузки JS происходит "гидратация" — статическая страница становится интерактивным SPA.

// Next.js (React) — страница автоматически рендерится на сервере export default function Page({ data }) { return <div>{data}</div>; }
  • Рендеринг: сервер (первый раз) + клиент (после гидратации).
  • SEO: отличное (как у MPA).
  • TTI: контент виден быстро (TTV — Time to View), но интерактивность наступает только после гидратации.
  • Плюсы: сочетание SEO и интерактивности.
  • Минусы: высокая нагрузка на сервер (CPU-intensive), риск "разрыва" между гидратацией и статикой.
📌 Примеры SSR: Next.js, Nuxt.js, Remix — используются для интернет-магазинов, новостных порталов с виджетами.

4. SSG (Static Site Generation)

Принцип: HTML-страницы генерируются один раз на этапе сборки (build-time). Пользователь получает готовый статический файл с CDN — максимальная скорость.

// Eleventy, Hugo, или Next.js (export) export default { async getStaticProps() { const data = await fetchData(); return { props: { data } }; } }
  • Рендеринг: заранее на сервере сборки (CI).
  • SEO: идеальное.
  • TTI: мгновенное (статика + минимум JS).
  • Плюсы: скорость, безопасность, дешевый хостинг (CDN).
  • Минусы: перегенерация при каждом изменении контента, не подходит для персонализированных данных.
📌 Примеры SSG: блоги, документация, лендинги, сайты-визитки (Hugo, Gatsby, Astro, 11ty).

🧠 Важные термины

  • TTV (Time to View) — время, когда пользователь увидел контент.
  • TTI (Time to Interactive) — время, когда можно кликать и взаимодействовать.
  • Гидратация (hydration) — процесс, при котором статический HTML, полученный с сервера, "оживляется" клиентским JS (вешаются обработчики событий, начинают работать хуки).
  • ISR (Incremental Static Regeneration) — усовершенствование SSG (Next.js), позволяющее обновлять статические страницы фоном без полной перегенерации.
