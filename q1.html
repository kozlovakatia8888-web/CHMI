Архитектурные парадигмы веб-фронтенда

MPA, SPA, SSR, SSG — как они влияют на рендеринг, SEO и время до интерактивности (TTI). Разбираем эволюцию веба.

01
MPA Multi‑Page App
Классический веб: каждый клик — полная перезагрузка страницы. Сервер отдаёт готовый HTML.
✅ Идеальное SEO · 🧠 Простота · 🚫 Медленная навигация
02
SPA Single‑Page App
Одна страница, роутинг на клиенте. Грузит JS-бандл, дальше — только данные.
⚡ Мгновенная навигация · 🐢 Плохой первый TTI · ❌ SEO требует ухищрений
03
SSR Server‑Side Rendering
Первый экран — серверный HTML, потом гидратация в SPA.
🎯 Быстрый TTV + SEO · ⚠️ Высокая нагрузка на сервер · 🧩 Сложность гидратации
04
SSG Static Site Gen
Генерация HTML на этапе сборки. Идеально для CDN и контентных сайтов.
🚀 Максимальная скорость · 🔒 Безопасность · 🔄 Не подходит для частых обновлений

🧠 Детали, которые важно понять

📌 TTV (Time to View) — когда пользователь видит контент. 📌 TTI (Time to Interactive) — когда может кликать и взаимодействовать.

В SSR возникает разрыв: страница отрисована быстро, но кнопки не работают, пока не загрузится JavaScript и не произойдёт гидратация. Это одна из главных проблем, которую решают Islands Architecture (частичная гидратация).

⚙️ Incremental Static Regeneration (ISR) — улучшенный SSG: страницы обновляются фоном после сборки, не требуя перегенерации всего сайта. Используется в Next.js.

// Пример: в SPA навигация без перезагрузки
// Роутер на клиенте меняет URL и подгружает данные fetch
// В SSG всё — готовый HTML, загружаемый почти мгновенно.

🗺️ Дорожная карта: как выбрать?

✅ Блог, документация, лендинг → SSG (максимум скорости) или MPA.

✅ Интернет-магазин / новостной портал с важным SEO → SSR (Next.js/Nuxt) или гибрид SSG + SSR.

✅ Личный кабинет, CRM, онлайн-редактор → SPA (интерактивность превыше SEO).

✅ Корпоративный сайт с редкими правками → MPA на Django/Rails или SSG.

🔥 Современный тренд: Astro (Islands) — по умолчанию SSG, но можно подключать интерактивные компоненты React/Vue изолированно.

🔍 А что с SEO для SPA?

SPA изначально плохо индексируются, потому что краулеры не исполняют JavaScript. Решения:

  • 🔹 Dynamic Rendering (рендеринг для ботов на сервере через Puppeteer).
  • 🔹 Prerendering (генерация статичных копий для популярных маршрутов).
  • 🔹 Использование SSR-фреймворков (Next.js, Nuxt) вместо голого SPA.
