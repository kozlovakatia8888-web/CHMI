🗄️ Объектно-реляционное несоответствие и ORM

Проблема impedance mismatch, как ORM (Hibernate, SQLAlchemy) решает её, и главные антипаттерны: N+1, божественная сущность, массовые операции.

⚠️ Проблема объектно-реляционного несоответствия

АспектОбъектная модельРеляционная модель
СтруктураСеть объектов, ссылкиТаблицы, примитивные типы
НаследованиеКласс → подклассыНет прямого способа
ИдентичностьРавенство vs ссылочнаяПервичный ключ
СвязиПрямые ссылки (user.orders)Внешние ключи (user_id)
// Объектный мир class User: id: int name: str orders: List[Order] // Реляционный: таблицы users (id,name) и orders (id,user_id,total)

🔧 Как ORM решает проблему (Hibernate, SQLAlchemy, EF)

  • Маппинг — связывание класса с таблицей через аннотации/декораторы.
  • Identity Map — гарантия, что один кортеж = один объект.
  • Ленивая загрузка (Lazy Loading) — подгрузка связанных объектов при первом обращении.
  • Жадная загрузка (Eager Loading) — явный JOIN для избежания N+1.
  • Dirty Tracking — отслеживание изменений для UPDATE только изменённых полей.
  • DSL-запросы — трансляция родного языка в SQL.

🧨 Антипаттерны ORM

  • N+1 запрос — ленивая загрузка в цикле. Решение: joinedload/selectinload.
  • Божественная сущность — класс со всеми полями → декомпозиция, DTO.
  • Массовые операции через цикл — тысячи UPDATE. Решение: массовые update.
  • Сериализация с ленивой загрузкой → LazyInitializationException. Решение: DTO или жадная загрузка.
  • Смешивание бизнес-логики с ORM → выносить в сервисы.
# Проблема N+1 users = session.query(User).all() # 1 запрос for user in users: print(user.orders) # N запросов # Решение users = session.query(User).options(joinedload(User.orders)).all()
