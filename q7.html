🗄️ Вопрос 7. Архитектура ORM

Объектно-реляционное несоответствие (impedance mismatch), как ORM решает проблему, и антипаттерны: N+1, божественная сущность, массовые операции.

⚠️ Объектно-реляционное несоответствие

ORM (Object-Relational Mapping) — технология, позволяющая работать с БД через объекты языка, минуя ручной SQL. Между объектной и реляционной парадигмами существует фундаментальное противоречие — impedance mismatch.

АспектОбъектная модельРеляционная модельНесоответствие
СтруктураСеть объектов, ссылкиЖёсткие таблицы, примитивные типыКак сохранить ссылку на объект?
НаследованиеКласс → подклассыНет прямого способаОдна таблица на иерархию? Таблица на подкласс?
ИдентичностьРавенство vs ссылочная идентичностьПервичный ключКак сопоставить объект в памяти и строку в БД?
СвязиПрямые ссылки (user.orders)Внешние ключи (user_id)Автоматически загружать связанные объекты?
Типы данныхСложные: enum, datetime, коллекцииПростые: INT, VARCHAR, DATEТребуется преобразование
// Пример: объектный мир class User: id: int name: str orders: List[Order] # ссылка на другие объекты class Order: id: int total: float # В БД: таблица users (id, name), таблица orders (id, user_id, total)

🔧 Как ORM решает проблему (Hibernate, SQLAlchemy, EF)

  • Маппинг (Mapping) — аннотации/декораторы, связывающие класс с таблицей и поля с колонками.
  • Identity Map — гарантирует, что один кортеж БД представлен одним объектом в памяти.
  • Ленивая загрузка (Lazy Loading) — связанные объекты подгружаются при первом обращении.
  • Жадная загрузка (Eager Loading) — явное указание загрузить связь через JOIN или отдельный запрос.
  • Dirty Tracking — отслеживание изменений полей для генерации UPDATE только по изменённому.
  • DSL-запросы — запросы на родном языке (HQL, LINQ, SQLAlchemy Core) транслируются в SQL.
# SQLAlchemy пример маппинга class User(Base): __tablename__ = 'users' id = Column(Integer, primary_key=True) name = Column(String) orders = relationship("Order", back_populates="user")

🧨 Антипаттерны использования ORM

1. Проблема N+1 запроса

Выборка N родителей (1 запрос), затем в цикле для каждого — подгрузка детей (N запросов). Итого N+1.

# Плохо: ленивая загрузка в цикле users = session.query(User).all() # 1 запрос for user in users: print(user.orders) # N запросов

Решение: жадная загрузка (joinedload / selectinload / JOIN FETCH).

users = session.query(User).options(joinedload(User.orders)).all() # 1-2 запроса

2. Божественная сущность (God Entity)

Один класс пытается отобразить слишком много таблиц и полей. Содержит десятки полей, половина из которых не нужна в большинстве сценариев.

Решение: декомпозиция на более мелкие классы (UserProfile, UserSettings, UserRelations), использование DTO/проекций.

3. Игнорирование N+1 в циклах

Разработчик не включает логирование SQL и не видит лавину запросов. Часто проявляется при выводе списков в веб-интерфейсах.

Решение: включить логирование SQL на этапе разработки, использовать профайлеры.

4. Массовые операции через цикл

Обновление тысяч записей через цикл с save() или update() каждого объекта — тысячи отдельных UPDATE.

# Плохо: цикл с коммитами for user in users: user.status = 'inactive' session.commit()
# Хорошо: массовый update session.query(User).filter(User.active == True).update({'status': 'inactive'}) session.commit()

5. Отсутствие жадной загрузки при сериализации в API

Связанные объекты подгружаются лениво уже во время сериализации, когда сессия ORM может быть закрыта. → LazyInitializationException или лишние запросы.

Решение: загружать всё необходимое до закрытия сессии, использовать DTO вместо сущностей.

6. Смешивание бизнес-логики и логики доступа к данным

ORM-сущности становятся «умными»: содержат валидацию, расчёты, отправку уведомлений. Нарушает разделение слоёв (Data Mapper).

Решение: оставить сущности просто хранилищами данных (DTO), бизнес-логику вынести в сервисы.
