🗄️ Архитектура ORM — объектно-реляционное отображение

Объектно-реляционное несоответствие, решения, антипаттерны N+1, God Object и массовые операции.

⚠️ Объектно-реляционное несоответствие (Impedance Mismatch)

Различия между объектно-ориентированной моделью и реляционной БД:

  • Структура: объекты — сеть ссылок, таблицы — плоские.
  • Наследование: в ООП есть подклассы, в БД нет (требуется стратегия маппинга).
  • Идентичность: объекты сравниваются по ссылке, строки — по первичному ключу.
  • Связи: объектные ссылки → внешние ключи.
  • Типы данных: сложные типы (enum, date) → простые (INT, VARCHAR).
ORM (Hibernate, SQLAlchemy) решает это с помощью маппинга, Identity Map, ленивой/жадной загрузки.

🧨 Антипаттерны ORM

  • N+1 запрос: выборка родителей (1 запрос) + в цикле подгрузка детей (N запросов). Решение: жадная загрузка (JOIN FETCH).
  • God Object (божественная сущность): один класс содержит все поля и связи, перегрузка данных. Решение: декомпозиция, DTO.
  • Массовые операции через цикл: обновление тысяч записей по одной (1000 UPDATE). Решение: массовый UPDATE.
  • Смешивание бизнес-логики с ORM: сущности содержат валидацию, отправку уведомлений. Решение: выносить логику в сервисы.
// Плохо: N+1 users = session.query(User).all() for user in users: print(user.orders) // каждый раз новый запрос // Хорошо: жадная загрузка users = session.query(User).options(joinedload(User.orders)).all()
