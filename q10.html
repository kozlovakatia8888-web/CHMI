🔑 Протокол OAuth 2.0 — углублённо

Роли участников, базовый поток Authorization Code Flow и решаемая проблема безопасности (делегирование доступа без пароля).

👥 Роли участников (4 стороны)

РольКто этоПример
Владелец ресурса (Resource Owner)Пользователь, у которого есть данные на сервере ресурсов.Вы, владелец фото в Google Фото.
Клиент (Client)Приложение, которое хочет получить доступ к данным пользователя.Сайт «Кинопоиск», мобильное приложение.
Сервер авторизации (Authorization Server)Служба, которая аутентифицирует пользователя, запрашивает согласие и выдаёт токены доступа.accounts.google.com, login.microsoftonline.com
Сервер ресурсов (Resource Server)Сервер, хранящий защищённые данные пользователя и принимающий запросы с токенами.photos.googleapis.com

🔐 Authorization Code Flow (базовый поток)

Пошагово:

  1. Клиент перенаправляет пользователя на сервер авторизации с параметрами: client_id, redirect_uri, response_type=code, scope, state.
  2. Пользователь аутентифицируется на сервере авторизации (если ещё не залогинен).
  3. Пользователь даёт согласие на доступ к запрошенным данным (scope).
  4. Сервер авторизации генерирует одноразовый авторизационный код и перенаправляет браузер на redirect_uri клиента с параметрами ?code=...&state=....
  5. Бэкенд клиента отправляет прямой запрос (сервер-сервер) на сервер авторизации, передавая code, client_id, client_secret, grant_type=authorization_code.
  6. Сервер авторизации проверяет код и секрет, возвращает access token (и опционально refresh token).
  7. Клиент использует access token для запросов к серверу ресурсов.
🔐 Почему код, а не сразу токен? Код передаётся через браузер (небезопасно), а токен обменивается по защищённому каналу сервер-сервер, что исключает перехват. Это ключевое улучшение безопасности по сравнению с устаревшим Implicit Flow.

🛡️ Какую проблему безопасности решает OAuth 2.0?

Проблема: приложение хочет доступ к данным пользователя на другом сервисе. Прямой способ — отдать свой пароль приложению. Это ужасно:

  • Приложение получает полный доступ к аккаунту.
  • Нельзя отозвать доступ, кроме как сменив пароль.
  • Если приложение скомпрометируют, злоумышленники украдут пароль.

Решение OAuth 2.0:

  • Вы даёте ограниченный доступ (scope email profile).
  • Вы не передаёте пароль — приложение получает токен.
  • Вы можете отозвать доступ в любой момент в настройках, не меняя пароль.
  • Токен имеет ограниченный срок жизни.
💡 OAuth 2.0 — это авторизация, не аутентификация. Для аутентификации поверх OAuth используют OpenID Connect (добавляет ID token).
