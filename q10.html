🔑 Вопрос 10. Протокол OAuth 2.0

Роли участников, базовый поток Authorization Code Flow и проблема безопасности, которую решает OAuth.

📌 Что такое OAuth 2.0

Протокол авторизации, который позволяет одному приложению (клиенту) получить ограниченный доступ к ресурсам пользователя на другом сервисе без передачи пароля. Пример: «Войти через Google» на стороннем сайте — сайт получает email и аватар, но никогда не узнает ваш пароль Google.

🎯 Решаемая проблема — безопасное делегирование доступа: дать приложению «ключ от сейфа» (доступ к данным), не отдавая главный ключ (пароль).

👥 Роли участников (4 стороны)

РольКто этоПример
Владелец ресурса
(Resource Owner) 		Пользователь, у которого есть данные на сервере ресурсов. Вы, владелец фото в Google Фото.
Клиент
(Client) 		Приложение, которое хочет получить доступ к данным пользователя. Сайт «Кинопоиск», мобильное приложение.
Сервер авторизации
(Authorization Server) 		Служба, которая аутентифицирует пользователя, запрашивает согласие и выдаёт токены доступа. accounts.google.com, login.microsoftonline.com
Сервер ресурсов
(Resource Server) 		Сервер, хранящий защищённые данные пользователя и принимающий запросы с токенами. photos.googleapis.com

Примечание: сервер авторизации и сервер ресурсов могут быть одним и тем же сервером (как у Google), но по спецификации это разные логические роли.

🔐 Базовый поток: Authorization Code Flow

Самый безопасный поток для веб-приложений с бэкендом. Включает этапы с временным кодом авторизации, который обменивается на токены на сервере клиента, не exposing токен в браузере.

1. Пользователь нажимает «Войти через Google». Клиент формирует URL для перенаправления на сервер авторизации с параметрами: client_id, redirect_uri, response_type=code, scope, state.
2. Браузер пользователя перенаправляется на сервер авторизации Google (страница входа).
3. Google аутентифицирует пользователя (если ещё не залогинен) и запрашивает согласие («Разрешить Кинопоиску доступ к email и аватару?»).
4. Пользователь нажимает «Разрешить». Сервер авторизации генерирует одноразовый авторизационный код и перенаправляет браузер обратно на redirect_uri клиента с параметрами ?code=...&state=....
5. Браузер попадает на redirect_uri клиента с кодом. Код виден в адресной строке, но короткоживущий (1–10 минут) и одноразовый.
6. Бэкенд клиента отправляет прямой запрос (сервер–сервер, не через браузер) на сервер авторизации Google, передавая code, client_id, client_secret, grant_type=authorization_code, redirect_uri.
7. Сервер авторизации проверяет код и секрет клиента, возвращает access token (и опционально refresh token). Бэкенд клиента использует access token для запросов к серверу ресурсов (например, для получения email).
💡 Почему code, а не сразу token? Код передаётся через браузер (небезопасная среда), а токен обменивается по прямому сервер-серверному каналу, который злоумышленник не может перехватить. Это ключевое улучшение безопасности по сравнению с устаревшим Implicit Flow.

🛡️ Какую проблему безопасности решает OAuth 2.0?

Проблема: Приложение хочет доступ к данным пользователя на другом сервисе. Прямой способ — отдать свой пароль этому приложению. Это ужасно:

  • Приложение получает полный доступ к аккаунту (почта, диск, календарь).
  • Нельзя отозвать доступ, кроме как сменив пароль (что повлияет на все другие приложения).
  • Если приложение скомпрометируют, злоумышленники украдут пароль.

Решение OAuth 2.0:

  • Вы даёте ограниченный доступ (scope email profile) — приложение не может читать почту или удалять файлы, только то, что разрешили.
  • Вы не передаёте пароль — приложение получает токен, выданный специально для него.
  • Вы можете отозвать доступ в любой момент в настройках, не меняя пароль.
  • Токен имеет ограниченный срок жизни (access token — часы, refresh token может быть отозван).
🎯 Дополнительные решаемые проблемы: делегирование без передачи учётных данных, минимизация прав (least privilege), разделение аутентификации и авторизации, защита от CSRF через параметр state.

⚠️ Важные замечания (чтобы не спутать)

  • OAuth 2.0 не решает аутентификацию — он не говорит, кто пользователь. Для аутентификации поверх OAuth 2.0 используют OpenID Connect (добавляет ID token).
  • Authorization Code Flow — основной для серверных веб-приложений. Для SPA без бэкенда используется Authorization Code Flow с PKCE (Proof Key for Code Exchange).
  • Refresh token — позволяет получить новый access token без участия пользователя, когда старый истёк. Хранится на сервере клиента.
  • Access token — короткоживущий (обычно 1 час), используется для запросов к Resource Server.
