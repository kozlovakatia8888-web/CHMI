🔑 Протокол OAuth 2.0

Роли участников, базовый поток Authorization Code Flow и решаемая проблема безопасности (делегирование доступа без пароля).

Роли участников

РольПример
Владелец ресурса (Resource Owner)Пользователь
Клиент (Client)Приложение (Кинопоиск)
Сервер авторизации (Authorization Server)accounts.google.com
Сервер ресурсов (Resource Server)API Google Photos

Authorization Code Flow (базовый поток)

1. Клиент перенаправляет пользователя на сервер авторизации с параметрами: client_id, redirect_uri, response_type=code, scope, state.

2. Пользователь аутентифицируется и даёт согласие.

3. Сервер авторизации возвращает временный код через редирект на redirect_uri.

4. Бэкенд клиента обменивает код + client_secret на access token по прямому сервер-серверному каналу.

5. Клиент использует access token для доступа к серверу ресурсов.

🔐 Почему код, а не сразу токен? Код передаётся через браузер (небезопасно), а токен — по защищённому каналу сервер-сервер, что исключает перехват.

Какую проблему решает OAuth 2.0?

Позволяет приложению получить ограниченный доступ к данным пользователя без передачи пароля. Пользователь может отозвать доступ в любой момент, не меняя пароль.

  • ✅ Ограниченные права (scopes).
  • ✅ Отзыв доступа без смены пароля.
  • ✅ Короткоживущие токены.
⚠️ OAuth 2.0 — это авторизация, не аутентификация. Для аутентификации поверх OAuth используют OpenID Connect.
