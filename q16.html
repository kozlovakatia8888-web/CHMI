🔒 Вопрос 16. Безопасность веб-приложений

XSS, CSRF, SQL-инъекции — описание атак и встроенные механизмы защиты в современных фреймворках.

1. XSS (Cross-Site Scripting) — межсайтовый скриптинг

Суть атаки: Внедрение вредоносного JavaScript-кода в веб-страницу, который выполняется в браузере жертвы. Кража куки, сессий, перенаправление, подмена контента.

Пример отражённого XSS:
https://site.com/search?q=<script>alert(document.cookie)</script> — если сервер выводит параметр без экранирования.
Тип XSSКак возникает
Отражённый (Reflected)Скрипт в URL — немедленно возвращается в ответе, жертву обманом заставляют перейти по ссылке.
Хранимый (Stored)Скрипт сохраняется в БД (комментарий, сообщение) и выполняется у всех, кто открывает страницу. Наиболее опасен.
DOM-basedУязвимость в клиентском JS (например, location.hash в innerHTML), сервер не при чём.

🛡️ Защита от XSS (что делают фреймворки)

  • Автоматическое экранирование в шаблонах: React (JSX экранирует в {}), Vue ({{ }}), Angular, Django ({{ var }}), Jinja2. Опасно: v-html, dangerouslySetInnerHTML, |safe.
  • Content Security Policy (CSP): заголовок, ограничивающий источники скриптов (script-src 'self').
  • HttpOnly cookie: запрет доступа к cookie через document.cookie (даже при XSS сессия не украдена).
  • Экранирование в нужном контексте: атрибуты, JS, URL — использовать специальные функции.

2. CSRF (Cross-Site Request Forgery) — подделка межсайтового запроса

Суть атаки: Злоумышленник заставляет браузер аутентифицированного пользователя отправить поддельный запрос на уязвимый сайт (например, перевод денег, смена пароля).

Классический пример:
<img src="http://bank.com/transfer?to=hacker&amount=1000" width="0" height="0">
При загрузке страницы с evil.com браузер отправит GET-запрос с куками жертвы.

🛡️ Защита от CSRF (встроенная в фреймворки)

  • CSRF-токены (синхронизатор токенов): сервер генерирует случайный токен, сохраняет в сессии, вставляет в каждую форму. При отправке проверяет. Атакующий сайт не может узнать токен. Фреймворки: Django ({% csrf_token %}), Spring Security, Rails, Laravel.
  • SameSite атрибут cookie: SameSite=Lax — cookie не отправляются при межсайтовых POST-запросах; SameSite=Strict — даже при переходах по ссылкам (защита максимальная, но может сломать переходы).
  • Проверка Referer/Origin: дополнительный барьер (но не надёжен).
  • Идемпотентность GET: GET-запросы не должны менять состояние (это уменьшает риск, но CSRF может использовать POST).

3. SQL-инъекции (SQL Injection)

Суть атаки: Внедрение SQL-кода в параметры запроса, который неправильно конкатенируется в SQL-запрос. Позволяет читать, изменять, удалять данные, выполнять команды на сервере.

Уязвимый код (Python):
query = f"SELECT * FROM users WHERE name = '{user_input}'"
При вводе ' OR '1'='1 возвращаются все пользователи.
'; DROP TABLE users; -- — удаление таблицы.

🛡️ Защита от SQL-инъекций (встроенная в фреймворки)

  • Параметризованные запросы (prepared statements): запрос и данные разделены. СУБД компилирует план, затем подставляет значения безопасно.
    Примеры: cursor.execute("SELECT * FROM users WHERE name = %s", (user_input,)) (Python), PreparedStatement (Java), SqlCommand с параметрами (C#).
  • ORM (Hibernate, SQLAlchemy, Django ORM, Entity Framework): по умолчанию генерируют параметризованные запросы. Опасность: raw(), execute() с конкатенацией.
  • Экранирование — не рекомендуется (легко ошибиться).
  • Минимизация прав БД: приложение подключается с минимальными привилегиями (без DROP, ALTER).
  • Валидация ввода как дополнительный слой (типы, длина, белый список).

📊 Сводная таблица: уязвимости и механизмы защиты

УязвимостьКраткое описаниеВстроенная защита во фреймворках
XSSВнедрение JS-кода в страницу. Автоматическое экранирование в шаблонах (React, Vue, Angular, Django, Jinja); CSP; HttpOnly cookie.
CSRFПринудительный запрос от аутентифицированного пользователя с другого сайта. CSRF-токены в формах (Django, Spring, Rails, Laravel); SameSite атрибут (Lax/Strict).
SQL-инъекцияВнедрение SQL-кода в запрос к БД. Параметризованные запросы / ORM (Hibernate, SQLAlchemy, Django ORM, EF). Никогда не конкатенировать пользовательский ввод.
💡 Главное правило: никогда не доверяйте пользовательскому вводу. Фреймворки дают защиту «из коробки», но разработчик должен сознательно её не отключать (|safe, dangerouslySetInnerHTML, raw SQL).

📌 Итог для экзамена (тезисы)

  • XSS — выполнение чужого JS в браузере. Защита: экранирование (фреймворки делают автоматически в шаблонах), CSP, HttpOnly.
  • CSRF — подделка запроса от лица пользователя. Защита: CSRF-токены (встроены в Django, Spring, Rails), SameSite=Lax/Strict.
  • SQL injection — внедрение SQL в запросы. Защита: параметризованные запросы / ORM (фреймворки используют их по умолчанию, если не писать raw SQL).
Ключевое: современные фреймворки решают эти проблемы автоматически при условии, что разработчик не использует обходные пути (сырые SQL, отключение экранирования, небезопасные атрибуты).
