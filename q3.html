📏 Количественные методы HCI: закон Фиттса и закон Хика

Как математика помогает проектировать удобные интерфейсы. Глубокий разбор с формулами, примерами и тестом.

1. Закон Фиттса (Fitts's Law)

Формулировка: Время, необходимое для быстрого перемещения указателя к цели, зависит от расстояния до цели и размера цели. Чем дальше и меньше цель — тем дольше.

T = a + b · log₂(D / W + 1)
  • T — среднее время достижения цели (ms)
  • D — расстояние от стартовой точки до центра цели
  • W — ширина цели в направлении движения
  • a, b — константы, зависящие от устройства ввода (например, для мыши a≈0, b≈0.1 с/бит)
  • log₂(D/W+1) — индекс сложности (ID) в битах
Пример: кнопка шириной 100px, расстояние 200px → ID = log₂(200/100+1) = log₂(3) ≈ 1.585 бит Время ≈ 0 + 0.1 * 1.585 ≈ 0.158 с = 158 мс (идеальное) Увеличим D до 400px, W=50px → ID = log₂(400/50+1) = log₂(9) ≈ 3.17 → время ≈ 317 мс

📌 Применение в UI

  • Большие кнопки для важных действий (например, «Оплатить»).
  • Близкое расположение частых элементов к курсору (контекстные меню, панели инструментов).
  • Использование краёв экрана — углы имеют бесконечный эффективный размер (закон «волшебных углов»). Кнопка «Пуск» в Windows, системный трей.
  • Мобильные интерфейсы: минимальная область нажатия 44×44pt, важные кнопки в нижней части (ближе к большому пальцу).
⚠️ Типичная ошибка: маленькая кнопка «Отправить» вдали от полей ввода, либо плавающая кнопка FAB в центре, далеко от края.

2. Закон Хика (Hick-Hyman Law)

Формулировка: Время принятия решения (выбора из нескольких вариантов) растёт логарифмически с увеличением количества альтернатив.

RT = a + b · log₂(n + 1)
  • RT — время реакции (ms)
  • n — количество равновероятных альтернатив
  • +1 — учёт опции «не выбирать ничего»
  • a, b — константы (b≈150 мс для человека)
Пример: выбор из 2 опций → log₂(3) ≈ 1.585 → RT ~ 1.585*150 ≈ 238 мс Выбор из 8 опций → log₂(9) ≈ 3.17 → RT ~ 3.17*150 ≈ 475 мс (всего в 2 раза дольше, а не в 4)

📌 Применение в UI

  • Группировка меню — не более 5–7 пунктов в одном меню, иначе разбивать на категории.
  • Прогрессивное раскрытие — сначала показывать основные варианты, дополнительные — по клику «Расширенные настройки».
  • Поиск и фильтры — вместо длинного списка использовать поиск с автодополнением.
  • Стандартные паттерны — пользователь не перебирает варианты, если знает, что «Файл → Сохранить» работает как обычно.
🎯 Итог: закон Хика не означает, что нужно избегать сложных интерфейсов, но требует иерархической организации и сокращения выбора на каждом шаге.

🚫 Частые ошибки проектирования

  • Слишком маленькие интерактивные элементы (ссылки 12px, чекбоксы без увеличенной области).
  • Длинные плоские списки без группировки (например, 30 пунктов настроек).
  • Кнопка действия далеко от места последнего клика (например, кнопка «Применить фильтр» вверху, а фильтры внизу).
  • Скрытые жесты без визуальных подсказок (пользователь не знает о существовании альтернативы).
