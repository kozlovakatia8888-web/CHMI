🛠️ Вопрос 18. Инструментарий бэкенд-разработки

Middleware и конвейер обработки запроса (pipeline). Dependency Injection (DI) — механизм, преимущества для тестируемости и гибкости.

🔁 Middleware и конвейер (pipeline) обработки запроса

Middleware — функции (или классы), которые участвуют в обработке входящего HTTP-запроса и исходящего ответа. Они вызываются последовательно, образуя конвейер. Каждая middleware может выполнить код до передачи запроса дальше, передать управление следующей (или завершить), а также выполнить постобработку.

// Express.js пример app.use(logger); // middleware 1 app.use(authentication); // middleware 2 app.use(compression); // middleware 3 app.get('/users', handler); // терминальный обработчик

Порядок вызова:
Запрос → [m1] → [m2] → [m3] → обработчик → [m3] → [m2] → [m1] → Ответ

// Пример middleware с next() app.use((req, res, next) => { console.log('Логирование запроса'); next(); // передаём дальше }); app.use((req, res, next) => { if (!req.headers.authorization) { return res.status(401).send('Unauthorized'); // прерывание цепочки } next(); }); app.get('/', (req, res) => res.send('Hello'));

📌 Где применяется в разных фреймворках

  • Express.js — app.use(), порядок важен.
  • Django — классы в MIDDLEWARE, методы process_request, process_response.
  • Spring Boot — фильтры (Filter), перехватчики (HandlerInterceptor).
  • ASP.NET Core — app.UseMiddleware<T>().
  • FastAPI — @app.middleware("http").
🎯 Типовые задачи middleware: логирование, аутентификация/авторизация, CORS, парсинг тела, сжатие, rate limiting, CSRF, добавление security-заголовков, трассировка.

💉 Внедрение зависимостей (Dependency Injection, DI)

Паттерн, при котором объект не создаёт свои зависимости сам (через new), а получает их извне (обычно через конструктор). Управление зависимостями выносится в DI-контейнер или явно в коде.

# Плохо (без DI) class UserService: def __init__(self): self.db = MySQLConnection() # жёсткая привязка self.logger = FileLogger()
# Хорошо (DI через конструктор) class UserService: def __init__(self, db_connection, logger): self.db = db_connection self.logger = logger # Композиция (снаружи) db = PostgreSQLConnection() logger = ConsoleLogger() user_service = UserService(db, logger)
Преимущество DIПояснение
ТестируемостьЛегко подставить mock-объекты вместо реальной БД или внешнего API.
Гибкость и переиспользованиеМеняем реализацию (например, MySQL → PostgreSQL) без изменения класса.
Слабая связанность (loose coupling)Класс знает только интерфейс зависимостей, а не их создание.
Упрощение конфигурацииDI-контейнер автоматически собирает граф объектов.

📦 DI-контейнеры и фреймворки

  • Spring (Java) — @Autowired, @Component.
  • ASP.NET Core — встроенный IServiceCollection.
  • Angular (фронтенд) — встроенный DI.
  • FastAPI (Python) — Depends.
  • Laravel, Symfony (PHP) — service container.
  • Django — нет встроенного, но есть библиотеки (django-injector, dishka).
// C# ASP.NET Core — регистрация и использование services.AddScoped<IUserRepository, UserRepository>(); services.AddScoped<UserService>(); public UserService(IUserRepository repo, ILogger<UserService> logger) { ... }
💡 Виды DI: constructor injection (предпочтительный), setter injection, interface injection.
⚠️ Антипаттерн — Service Locator (ServiceLocator.get('db')) — скрывает зависимости, усложняет тестирование.

📌 Итог для экзамена (тезисы)

ПонятиеСутьПримеры
Middleware Функции в цепочке обработки запроса (предобработка → обработчик → постобработка). Можно прервать или передать дальше. Express app.use(), Django MIDDLEWARE, ASP.NET Core app.Use().
Pipeline Последовательность вызова middleware (порядок регистрации важен). req → logger → auth → handler → auth → logger → res
Dependency Injection Объект получает зависимости извне (конструктор), не создаёт их сам. UserService(db, logger) вместо self.db = MySQLConnection().
Преимущества DI Тестируемость (mock), гибкость (смена реализации), слабая связанность, переиспользование. В тестах — MockDB(), в проде — PostgresDB().
DI-контейнер Фреймворк, автоматически создающий и предоставляющий зависимости. Spring @Autowired, ASP.NET Core IServiceCollection, FastAPI Depends.
🎯 Главный вывод: Middleware управляет потоком HTTP-запросов, разделяя сквозные задачи. DI делает код слабо связанным, тестируемым и легко конфигурируемым. Оба механизма — стандарт современной бэкенд-разработки.
