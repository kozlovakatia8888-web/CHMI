🧩 Вопрос 17. Микрофронтенды

Архитектура, способы интеграции (iframe, Web Components, Module Federation) и проблемы, которые решает этот подход.

📌 Что такое микрофронтенды?

Архитектурный подход, при котором фронтенд-приложение разбивается на небольшие, независимо разрабатываемые и развёртываемые части (микро-приложения), каждая из которых отвечает за свою бизнес-область. Аналог микросервисов на клиентской стороне.

🔍 Какие проблемы решает архитектура микрофронтендов?

  • Масштабирование разработки: каждая команда владеет своей областью (каталог, корзина, личный кабинет), нет «merge hell».
  • Независимое развёртывание: обновление одного микрофронтенда без пересборки всего приложения.
  • Технологическая гетерогенность: можно мигрировать постепенно (AngularJS → React), использовать разные фреймворки на одной странице.
  • Изоляция ошибок: падение одного микрофронтенда не «роняет» всю страницу.
  • Ускорение загрузки: пользователь получает только нужные части (ленивая загрузка).

🧩 Способы интеграции микрофронтендов

1. iframe (самый старый, максимальная изоляция)

Как работает: каждый микрофронтенд встраивается через <iframe>, обмен данными через postMessage.

✅ Плюсы: полная изоляция CSS/JS, не требует специальной сборки, можно встроить любое legacy-приложение.
❌ Минусы: проблемы с SEO, навигацией (история браузера), размером (подгонка под контент), дублирование зависимостей, плохая производительность.

2. Web Components (стандартный, технологически нейтральный)

Как работает: каждый микрофронтенд — кастомный HTML-элемент с Shadow DOM (изоляция стилей). Родитель вставляет тег <micro-cart>.

<micro-frontend-cart user-id="123"></micro-frontend-cart>
✅ Плюсы: родной стандарт, изоляция стилей (Shadow DOM), независимость от фреймворков, легко интегрировать.
❌ Минусы: поддержка браузеров (полифиллы), Shadow DOM мешает глобальным стилям, сложный SSR, дублирование зависимостей.

3. Module Federation (Webpack 5) — самый современный

Как работает: плагин Webpack, позволяющий динамически загружать модули из другого независимо собранного приложения во время выполнения. Общие библиотеки (React, lodash) можно сделать синглтонами — они загрузятся один раз.

// webpack.config.js (хост) new ModuleFederationPlugin({ name: 'host', remotes: { cart: 'cart@http://localhost:3001/remoteEntry.js' }, shared: { react: { singleton: true }, 'react-dom': { singleton: true } } });
✅ Плюсы: нулевое дублирование зависимостей, производительная ленивая загрузка, бесшовная навигация (общий роутер), возможность шарить состояние (Redux).
❌ Минусы: привязка к Webpack (есть аналоги для Vite), сложная настройка, возможны проблемы с версиями общих библиотек.

📌 Итог для экзамена (тезисы)

  • Микрофронтенды — разделение фронтенда на независимые части (каждая своя команда, свой стек, своё развёртывание).
  • Решаемые проблемы: масштабирование разработки, независимые релизы, технологическая гетерогенность, изоляция ошибок.
  • iframe — прост, полная изоляция, но проблемы с навигацией, SEO, дублирование зависимостей. Годится только для сторонних виджетов.
  • Web Components — стандарт, изоляция стилей (Shadow DOM), технологическая нейтральность, но дублирование зависимостей и сложности с SSR.
  • Module Federation (Webpack 5) — современный, общие зависимости, ленивая загрузка, бесшовная навигация. Рекомендуемый выбор для новых проектов, но сложный в настройке.
💡 Дополнительно: Single-SPA (метафреймворк для координации SPA), встраивание через NPM-пакеты (теряется независимое развёртывание).
