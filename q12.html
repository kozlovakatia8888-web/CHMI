🌐 Вопрос 12. Сетевые протоколы для API

REST, GraphQL и gRPC — сравнение, сильные и слабые стороны, сценарии использования.

📡 REST (Representational State Transfer)

Использует HTTP-методы (GET, POST, PUT, PATCH, DELETE) для операций над ресурсами, каждый ресурс идентифицируется URL. Данные обычно в JSON, сервер stateless.

GET /users/123 → возвращает пользователя GET /users/123/orders → возвращает заказы POST /orders → создаёт новый заказ

✅ Сильные стороны REST

  • Простота и привычность, отладка через curl/браузер.
  • Кеширование через HTTP-заголовки (Cache-Control, ETag).
  • Масштабируемость (stateless).
  • Понятная документация через OpenAPI.

❌ Слабые стороны REST

  • Over‑fetching (лишние поля) и under‑fetching (несколько запросов).
  • Фиксированная структура ответа.
  • Версионирование приводит к дублированию.
  • Несколько round‑trip'ов для сложных данных.
🎯 Сценарии: публичные API, CRUD-приложения, проекты, где важно кеширование и простота.

🔮 GraphQL

Единый endpoint (/graphql). Клиент отправляет запрос, явно указывая нужные поля. Сервер возвращает ровно запрошенные данные, включая связанные ресурсы.

query { user(id: 123) { name avatarUrl orders(limit: 5) { total items { name } } } }

✅ Сильные стороны GraphQL

  • Нет over‑fetching и under‑fetching.
  • Один запрос вместо многих (важно для мобильных сетей).
  • Сильная типизация (схема), интроспекция, инструменты (GraphiQL).
  • Удобная эволюция API (добавление полей без ломки клиентов).
  • DataLoader для объединения N+1 на сервере.

❌ Слабые стороны GraphQL

  • Сложность реализации на сервере (резолверы, защита от глубоких запросов).
  • Плохое кеширование HTTP (один URL).
  • Уязвимость к resource exhaustion (лимит глубины, сложности).
  • Не подходит для простых CRUD (оверхед).
🎯 Сценарии: сложные клиентские приложения (дашборды, соцсети), мобильные приложения, агрегация данных из разных источников.

⚡ gRPC

Фреймворк от Google на HTTP/2 и Protocol Buffers. Описывается .proto-файл, генерируется код клиента и сервера. Поддерживает стриминг.

service UserService { rpc GetUser (GetUserRequest) returns (User); rpc ListOrders (ListOrdersRequest) returns (stream Order); }

✅ Сильные стороны gRPC

  • Высокая производительность (бинарный Protobuf, HTTP/2).
  • Строгая типизация, ошибки на этапе компиляции.
  • Потоковая передача (server, client, bidirectional).
  • Автоматическая генерация кода на 10+ языках.
  • Встроенные механизмы: интерцепторы, балансировка, deadlines.

❌ Слабые стороны gRPC

  • Сложность отладки (бинарный формат, нужны grpcurl/grpcui).
  • Не поддерживается браузерами напрямую (требуется grpc-web).
  • Нет гибкости запроса — изменение полей требует изменения .proto.
  • Кривая обучения (Protobuf, HTTP/2).
🎯 Сценарии: внутренние микросервисы, системы реального времени (стриминг), высоконагруженные сервисы, полиглотные среды.

🧠 Итог: выбор протокола по сценарию

СценарийРекомендацияПочему
Публичное API для сторонних разработчиковRESTПростота, кеширование, документация, curl.
Внутренние микросервисы с высокой нагрузкойgRPCПроизводительность, типизация, стриминг.
Мобильное/веб-приложение со сложными экранамиGraphQLОдин запрос, клиент сам выбирает поля, меньше трафика.
Простое CRUD-приложение (блог, каталог)RESTОверхед GraphQL/gRPC не оправдан.
Агрегатор нескольких сервисов (API Gateway)GraphQL или gRPC GatewayGraphQL хорошо собирает данные, gRPC даёт производительность.
Система реального времени (чат, мониторинг)gRPC (streaming)Двунаправленный стриминг из коробки.
💡 Коротко: REST — золотая середина для большинства. GraphQL — когда клиент сложный и капризный. gRPC — когда нужно максимально быстро и строго внутри своей инфраструктуры.
