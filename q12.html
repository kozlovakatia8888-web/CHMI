🌐 Сетевые протоколы для API: REST, GraphQL, gRPC — углублённо

Сравнение, сильные и слабые стороны, сценарии использования, глоссарий.

📡 REST (Representational State Transfer)

Принцип: архитектурный стиль, использующий HTTP-методы (GET, POST, PUT, PATCH, DELETE) для операций над ресурсами. Каждый ресурс идентифицируется URL. Данные обычно в JSON. Сервер stateless.

GET /users/123 → возвращает пользователя GET /users/123/orders → возвращает заказы пользователя POST /orders → создаёт новый заказ
  • ✅ Плюсы: простота и привычность, кеширование через HTTP-заголовки, масштабируемость, документация через OpenAPI.
  • ❌ Минусы: over‑fetching (лишние поля) и under‑fetching (несколько запросов), фиксированная структура ответа, версионирование приводит к дублированию.
🎯 Сценарии: публичные API, CRUD-приложения, проекты с важным кешированием.

🔮 GraphQL

Принцип: единый endpoint (/graphql). Клиент отправляет запрос (query), явно указывая нужные поля. Сервер возвращает ровно запрошенные данные, включая связанные ресурсы за один запрос.

query { user(id: 123) { name avatarUrl orders(limit: 5) { total items { name } } } }
  • ✅ Плюсы: нет over‑fetching и under‑fetching, один запрос вместо многих, сильная типизация (схема), удобная эволюция API (добавление полей без ломки клиентов).
  • ❌ Минусы: сложность реализации на сервере (резолверы, защита от глубоких запросов), плохое кеширование HTTP, уязвимость к resource exhaustion.
🎯 Сценарии: сложные клиентские приложения (дашборды, соцсети), мобильные приложения, агрегация данных из разных источников.

⚡ gRPC

Принцип: фреймворк от Google на HTTP/2 и Protocol Buffers. Описывается .proto-файл, генерируется код клиента и сервера. Поддерживает стриминг (server, client, bidirectional).

service UserService { rpc GetUser (GetUserRequest) returns (User); rpc ListOrders (ListOrdersRequest) returns (stream Order); }
  • ✅ Плюсы: высокая производительность (бинарный Protobuf, HTTP/2), строгая типизация, стриминг, автоматическая генерация кода на 10+ языках.
  • ❌ Минусы: сложность отладки (бинарный формат), не поддерживается браузерами напрямую, нет гибкости запроса (изменение полей требует изменения .proto).
🎯 Сценарии: внутренние микросервисы, системы реального времени (стриминг), высоконагруженные сервисы.
