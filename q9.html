🔐 Аутентификация, сессии и JWT

Идентификация, аутентификация, авторизация — разница. Сравнение stateful (сессии) и stateless (JWT) подходов.

Идентификация, аутентификация, авторизация

ТерминВопросПример
Идентификация«Кто ты?»Ввод логина alice@example.com
Аутентификация«Докажи»Ввод пароля, проверка хеша
Авторизация«Что можно?»Проверка роли перед удалением

Сессии (stateful)

Сервер хранит сессию (в памяти/Redis). Клиенту выдаётся Session ID в cookie. При каждом запросе сервер ищет сессию по ID.

// Установка сессии (Express) req.session.userId = user.id;
  • ✅ Плюсы: мгновенный отзыв доступа, защита от XSS (HttpOnly), простота.
  • ❌ Минусы: нагрузка на сервер, сложность масштабирования, уязвимость к CSRF.

JWT (stateless)

Сервер не хранит токен. Подписанный JSON (header.payload.signature) содержит данные о пользователе и срок истечения. Клиент хранит токен (localStorage) и отправляет в заголовке Authorization.

// Пример payload { "userId": 123, "role": "admin", "exp": 1735689600 }
  • ✅ Плюсы: масштабируемость, не нужна синхронизация сессий, подходит для микросервисов и мобильных приложений.
  • ❌ Минусы: нельзя отозвать до истечения, больший размер, уязвимость к XSS при хранении в localStorage.

Сравнение

ХарактеристикаСессии (stateful)JWT (stateless)
Хранение состоянияНа сервереВ токене
Отзыв доступаМгновенныйСложный (короткий TTL + refresh)
Защита от XSSХорошая (HttpOnly)Плохая (localStorage)
CSRFТребует токеновНе требуется (токен в заголовке)
💡 Выбор: классический веб → сессии; высоконагруженные API, микросервисы → JWT с коротким временем жизни.
