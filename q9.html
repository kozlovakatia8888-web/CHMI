🔐 Вопрос 9. Аутентификация, сессии и JWT

Идентификация, аутентификация, авторизация — разница. Сравнение stateful (сессии) и stateless (JWT) подходов.

🆔 Идентификация, аутентификация, авторизация — в чём разница?

ТерминВопросЧто происходитПример
Идентификация«Кто ты?»Пользователь сообщает идентификатор (логин, email).Ввод alice@example.com в поле логина.
Аутентификация«Докажи, что ты тот?»Проверка подлинности (пароль, биометрия, токен).Ввод пароля; сервер сверяет хеш.
Авторизация«Что тебе разрешено?»Определение прав доступа (роли, разрешения).Проверка, может ли alice удалить пост (роль reader → 403).
✈️ Аналогия с аэропортом: идентификация — показал паспорт; аутентификация — сверка фото и подлинности; авторизация — пускают в бизнес-зал только при наличии билета бизнес-класса.

🗄️ Подход 1: Состояние на сервере (сессии)

Как работает: после успешного входа сервер создаёт сессию (в памяти, Redis, БД), генерирует случайный Session ID, отправляет его клиенту в cookie (обычно HttpOnly, Secure). Браузер автоматически возвращает Session ID, сервер восстанавливает контекст пользователя.

✅ Плюсы сессий

  • Полный контроль на сервере — мгновенный отзыв доступа (удалить сессию).
  • Сессия не содержит данных пользователя, только случайный ID — безопасно при краже ID (но возможен угон).
  • Не требуется подпись (как в JWT), невозможно подделать ID.
  • Простая инвалидация (session.destroy()).
  • Можно хранить большие объёмы данных (корзина покупок).

❌ Минусы сессий

  • Нагрузка на сервер — хранилище сессий (Redis/БД) при миллионах пользователей.
  • Проблемы горизонтального масштабирования (липкие сессии).
  • Зависимость от cookie — неудобно для мобильных приложений и ботов.
  • Уязвимость к CSRF (нужны csrf-токены).
  • При перезапуске сервера сессии в памяти пропадают.

🎫 Подход 2: Stateless на основе JWT (JSON Web Token)

Как работает: сервер не сохраняет токен. При успешном входе создаётся JWT: header.payload.signature (подпись секретным ключом). Клиент хранит токен (обычно в localStorage) и отправляет в заголовке Authorization: Bearer <token>. Сервер проверяет подпись и извлекает данные (user_id, role, exp).

// Пример декодированного payload JWT { "user_id": 123, "role": "admin", "exp": 1735689600 }

✅ Плюсы JWT

  • Масштабируемость — любой сервер проверяет подпись без общего хранилища.
  • Нет нагрузки на хранилище сессий.
  • Работает с мобильными приложениями, микросервисами, любым клиентом (не зависит от cookie).
  • Самодостаточен — снижает количество запросов к БД.
  • Явное истечение (поле exp).

❌ Минусы JWT

  • Нельзя отозвать до истечения срока — украденный токен действует, пока не истечёт. Решение: короткое время жизни (15 мин) + refresh token в БД.
  • Большой размер (килобайты на каждый запрос).
  • Payload только подписан, не зашифрован (любой может прочитать содержимое). Нельзя класть пароли, карты.
  • Проблема с обновлением данных пользователя (роль изменилась — старый токен остаётся валидным).
  • Уязвимость к XSS при хранении в localStorage (токен доступен скрипту).
  • Нет встроенного механизма принудительного выхода (logout).

📌 Итог для экзамена (тезисы)

  • Идентификация — назвал себя. Аутентификация — доказал. Авторизация — проверка прав.
  • Сессии (stateful): сервер хранит состояние, Session ID в cookie. Плюсы: мгновенный отзыв, защита от XSS (HttpOnly). Минусы: нагрузка на сервер, сложность масштабирования, CSRF.
  • JWT (stateless): сервер не хранит ничего, токен подписан. Плюсы: масштабируемость, независимость от cookie, подходит для микросервисов. Минусы: нельзя отозвать, данные читаемы, уязвимость к XSS.
  • Выбор: классический веб (банки, админки) → сессии + CSRF-токены. Высоконагруженные API, микросервисы → JWT с коротким временем жизни и refresh token.
